Here’s the thread lifecycle in one-line explanations for each state:
New → Thread is created but start() has not been called yet.
Runnable → Thread is ready to run and waiting for CPU scheduling.
Running → Thread is actively executing its run() method.


Blocked / Waiting → Thread is paused, waiting for a lock, resource, or another thread’s action.


Terminated → Thread has finished execution or stopped.


👉 In short: Created → Ready → Executing → Paused (if needed) → Finished.
--------------------
- A critical section is a part of code where shared resources (like variables, files, objects) are accessed and must not be 
executed by more than one thread at the same time.

Synchronized method locks the entire method, whereas synchronized block locks only a specific part of the code.
--------------------------------------------------------------------------
| Aspect                | Instance Lock      | Class Lock        |
| --------------------- | ------------------ | ----------------- |
| Lock scope            | Object level       | Class level       |
| Lock object           | `this`             | `ClassName.class` |
| Shared across objects | ❌ No               | ✅ Yes             |
| Parallel execution    | ✅ Possible         | ❌ Not possible    |
| Used with             | Non-static methods | Static methods    |

Instance lock and class lock are completely different

✔️ They do NOT block each other

synchronized(this) { }          // instance lock
synchronized(MyClass.class) { } // class lock

👉 Both can run at the same time ✔️

5. When to Use
✅ Instance Lock
When data is object-specific
✅ Class Lock
When data is shared across all objects (static data)

“Intrinsic locks are JVM-managed object-level locks used by synchronized to provide mutual exclusion.
👉 An intrinsic lock (monitor) is a built-in lock every object has, used by synchronized to ensure only one thread accesses a critical section at a time.
Key Features
🔸 1. Built-in (No need to create)
Automatically available for every object
🔸 2. Mutual Exclusion
Only one thread at a time
🔸 3. Reentrant
Same thread can acquire lock multiple times
🔸 4. Used with synchronized
Behind the scenes, synchronized uses intrinsic locks

---------------------------------------------------------------------------------------------
synchronized is simple and JVM-managed, while ReentrantLock is more flexible and provides advanced locking features.

1️⃣ Fairness
👉 Controls which thread gets the lock first
✔️ Fair Lock
	ReentrantLock lock = new ReentrantLock(true);
	Threads get lock in FIFO order
	Prevents starvation
	
- Thread can be interrupted while waiting for lock	
- “ReentrantLock provides advanced features like fairness, tryLock, and interruptible locking, which are not available in synchronized.”

Visibility 👉 When one thread changes a variable, other threads can immediately see the updated value.
Atomicity 👉 An operation is executed completely or not at all—no intermediate state is visible to other threads.

🔥 Step-by-Step Approach
🔹 1. Identify the Symptom

First understand what’s wrong:
	High CPU 🔥
	Application hang / freeze ❌
	Slow response ⏳
	Threads not completing
	
	
🔹 2. Take Thread Dumps (Most Important)

👉 Capture 2–3 thread dumps at intervals (5–10 sec apart)

jstack <pid> > dump1.txt
jstack <pid> > dump2.txt

✔️ Multiple dumps help detect patterns

🔹 3. Analyze Thread Dumps
🔍 Check Thread States
		BLOCKED → waiting for lock 🔒 (lock contention)
		WAITING / TIMED_WAITING → waiting (wait(), sleep())
		RUNNABLE → may indicate high CPU or infinite loop

🔍 Look for Deadlocks
		👉 jstack shows directly:
		Found one Java-level deadlock:
		Thread-1 waiting for lock held by Thread-2
		Thread-2 waiting for lock held by Thread-1
🔍 Identify Patterns
		Same threads stuck in all dumps → real issue
		Threads stuck on same line → hotspot
		
🔹 4. Use Monitoring Tools
👉 VisualVM
		Threads tab → see live thread states
		CPU usage → find heavy threads
		Detect deadlocks visually
		
🔹 5. Correlate with Code
		Match stack trace → exact line number
		Identify:
		Synchronized blocks
		Infinite loops
		Blocking I/O
		
🔹 6. Common Issues You’ll Find
		Deadlock
		Livelock
		Thread starvation
		Excessive synchronization
		Infinite loops
		Blocking external calls
		
🔹 7. Fix Strategies
		Fix lock ordering (avoid deadlock)
		Reduce synchronization scope
		Replace with:
		Concurrent collections
		Async processing
		Optimize loops / logic

🔥 Key Ways to Make a Method Thread-Safe
🔹 1. Avoid Shared Mutable State (Best Approach ✅)
	👉 Use local variables (thread-confined)
	public int calculate(int a, int b) {
		int result = a + b; // local variable → thread-safe
		return result;
	}

✔️ No shared data → no issue
🔹 2. Use Synchronization

👉 Protect critical section

public synchronized void increment() {
    count++;
}
or
synchronized(this) {
    count++;
}

✔️ Only one thread at a time
🔹 3. Use Locks (Advanced)
👉 Use:
	ReentrantLock
	lock.lock();
	try {
		count++;
	} finally {
		lock.unlock();
	}

✔️ More control than synchronized

🔹 4. Use Immutable Objects
👉 Make objects read-only
final String name = "Neelam";
✔️ No modification → inherently thread-safe

🔹 5. Use Atomic Classes
👉 Instead of manual synchronization
AtomicInteger count = new AtomicInteger(0);
count.incrementAndGet();
✔️ Thread-safe without locks

🔹 6. Use Concurrent Collections
👉 Replace:
HashMap → ConcurrentHashMap
✔️ Built-in thread safety
🔹 7. Use ThreadLocal (When Needed)
👉 For thread-specific data
✔️ Avoid sharing altogether
🔹 8. Minimize Critical Section
👉 Lock only necessary code
✔️ Better performance

🔹 Important Interview Insight (6+ Years)
👉 Best practice order:
	Avoid sharing
	Use immutability
	Use concurrent utilities
	Use synchronization (last resort)
