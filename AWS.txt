What is a Security Group in AWS?

In Amazon Web Services, a Security Group is a virtual firewall that controls inbound and outbound traffic to your resources like:
	Amazon EC2
	Amazon RDS
	Amazon ECS

Simple Definition
	A Security Group controls who can access your AWS resource and on which port.

How It Works

Security Groups contain rules:
️⃣ Inbound Rules (Incoming Traffic)
	Controls who can connect to your server.
		Example:
		Allow HTTP → Port 80 → From anywhere (0.0.0.0/0)   // means Allow access from any IP address in the world
		Allow SSH → Port 22 → Only from your IP  // SSH is used to log in to your server.  

2️⃣ Outbound Rules (Outgoing Traffic)
	Controls where your server can send data.

You must allow:

| Type       | Port | Purpose                  |
| ---------- | ---- | ------------------------ |
| HTTP       | 80   | Website access           |
| Custom TCP | 8080 | Spring Boot default port |
| SSH        | 22   | Server login             |

Without opening port 8080 → your app will not be accessible.

Important Characteristics
	✔ Security Groups are stateful
	→ If inbound traffic is allowed, response traffic is automatically allowed.
	✔ They work at instance level
	✔ You can attach multiple instances to one security group

Interview Answer (Short Version)
	A Security Group in AWS is a stateful virtual firewall attached to resources like EC2 that controls inbound and outbound traffic 
	based on defined rules such as port number, protocol, and IP address.

Difference: Security Group vs NACL (Very Important Interview Question)
| Security Group          | NACL                  |
| ----------------------- | --------------------- |
| Works at instance level | Works at subnet level |
| Stateful                | Stateless             |
| Allows rules only       | Allows & Denies rules |

3️⃣ Create S3 Configuration Bean
@Configuration
public class AwsS3Config {

    @Value("${cloud.aws.region}")
    private String region;

    @Value("${aws.accessKey}")
    private String accessKey;

    @Value("${aws.secretKey}")
    private String secretKey;

    @Bean
    public S3Client s3Client() {
        return S3Client.builder()
            .region(Region.of(region))
            .credentialsProvider(
                StaticCredentialsProvider.create(
                    AwsBasicCredentials.create(accessKey, secretKey)
                )
            )
            .build();
    }
}

4️⃣ Create Service to Upload Bulk Images
@Service
public class BulkImageUploadService {

    @Autowired
    private S3Client s3Client;

    @Value("${aws.s3.bucket}")
    private String bucketName;

    public List<String> uploadImages(List<MultipartFile> files) {
        List<String> uploadedUrls = new ArrayList<>();

        for (MultipartFile file : files) {
            try {
                String key = "products/" + System.currentTimeMillis() + "-" + file.getOriginalFilename();

                PutObjectRequest request = PutObjectRequest.builder()
                        .bucket(bucketName)
                        .key(key)
                        .acl(ObjectCannedACL.PUBLIC_READ)   // Optional: Makes public
                        .contentType(file.getContentType())
                        .build();

                s3Client.putObject(request, RequestBody.fromBytes(file.getBytes()));

                String fileUrl = "https://" + bucketName + ".s3.amazonaws.com/" + key;
                uploadedUrls.add(fileUrl);

            } catch (IOException e) {
                throw new RuntimeException("Error uploading file: " + file.getOriginalFilename());
            }
        }

        return uploadedUrls;
    }
}

5️⃣ Expose REST API for Bulk Upload
@RestController
@RequestMapping("/api/images")
public class ImageUploadController {

    @Autowired
    private BulkImageUploadService uploadService;

    @PostMapping("/bulk")
    public ResponseEntity<List<String>> uploadMultipleFiles(
            @RequestParam("files") List<MultipartFile> files) {

        List<String> uploadedUrls = uploadService.uploadImages(files);
        return ResponseEntity.ok(uploadedUrls);
    }
}

Why?
Because:
Stage 1 → build the application
Stage 2 → run the application
This keeps the final image smaller.	

Step 1
FROM eclipse-temurin:17-jdk-alpine AS builder

Docker starts a container using Java 17 JDK Alpine image.

Step 2
WORKDIR /build

Docker creates and moves to this directory inside the container:

Step 3
COPY . .
This copies your entire project folder from the host machine into /build.

Example host project
project/
 ├── pom.xml
 ├── mvnw
 ├── src/
 └── target/

Inside container it becomes:
/build
 ├── pom.xml
 ├── mvnw
 ├── src/
 └── target/ (if already exists)


2️⃣ When does the .jar file appear?
After this command:
	RUN ./mvnw clean package -DskipTests
Maven builds the application and creates the .jar file.
Result:
/build/target/backend-0.0.1-SNAPSHOT.jar
----------------------
Q1: Why use jre instead of jdk in runtime stage?
- Because the application is already compiled.
- JRE contains only runtime libraries needed to run the application, making the image smaller and more secure.
-------------------
Q4: What is the benefit of multi-stage builds?
	Multi-stage builds separate build environment (JDK) and runtime environment (JRE), which reduces final image size and improves security.
